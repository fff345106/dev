## 关键发现

### 1. 管理员登录是最大瓶颈

- RPS 仅 **18**，是所有端点中最低的
- 平均延迟 **1,030ms**，是其他端点的 5-10 倍
- 510 次 socket 超时
- **根因**: 密码哈希验证（bcrypt）计算密集，在高并发下严重阻塞

**建议**:
- 引入登录速率限制（rate limiting）
- 考虑缓存密码验证结果（短期 session cache）
- 异步化密码哈希计算

### 2. 写入端点稳定性不足

- `创建草稿` 和 `提交审核` 出现 socket 超时和非 2xx 响应
- 高并发写入时数据库连接池可能成为瓶颈

**建议**:
- 检查 HikariCP 连接池配置（默认 10 连接可能不够）
- 增加 `spring.datasource.hikari.maximum-pool-size` 到 20-30
- 为写入操作添加请求队列或限流

### 3. 查询端点性能优秀

- 读取端点普遍在 500-1400 RPS 范围
- 统计信息和排行榜端点表现最佳（1,000+ RPS）
- 平均延迟均在 200ms 以内

### 4. 分页查询略慢于单条查询

- 纹样列表（分页）614 RPS vs 纹样详情（单条）1,179 RPS
- 差异约 2 倍，属于正常范围

## 优化建议优先级

### P0 - 紧急
1. **管理员登录性能优化** - 当前 18 RPS 无法支撑生产环境
2. **数据库连接池扩容** - 解决写入端点超时问题

### P1 - 重要
3. **添加全局限流** - 保护后端服务不被压垮
4. **监控告警** - 集成 Prometheus + Grafana 监控 RPS 和延迟

### P2 - 改善
5. **读写分离** - 将读取流量分流到只读副本
6. **缓存层** - 为热点数据（统计、排行榜）添加 Redis 缓存
7. **CDN 加速** - 纹样图片走 CDN 减轻服务器压力
